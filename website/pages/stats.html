🔗 How the Stats Pipeline Works

The backend follows a three-layer MVC pattern. Every interaction recorded on the sport pages passes through all three layers before appearing here.

▶ Write path: recording an interaction

  1. A user presses a button (e.g. Shoot on basketball.html). The sport page calls fetch('../api.php', { method: 'POST', body: JSON.stringify({ sport: 'basketball', type: 'shoot' }) }).
  2. api.php (the thin entry point) receives the POST, reads the JSON body, and calls $controller->apiInsertData($sport, $type).
  3. Controller (application/controller/controller.php) validates sport against an allowlist ['tennis', 'basketball', 'football'] and type against ['hit', 'juggle', 'shoot', 'dribble', 'swing']. Invalid values are rejected with a 400 response, no raw input ever reaches the database.
  4. Model (application/model/model.php) runs a PDO prepared statement: INSERT INTO interactions (sport, type) VALUES (:sport, :type). The row gets an auto-incremented id and a created_at timestamp from SQLite.
  5. api.php returns {"status":"ok","id":N} to the sport page. The sport page ignores the response (fire-and-forget).

▶ Read path: loading this page

  1. On load (and every 5 seconds), stats.html calls fetch('../api.php?action=stats').
  2. api.php sees action=stats and calls $controller->apiGetStats().
  3. Controller delegates directly to $model->dbGetStats(): no validation needed for a read.
  4. Model runs three SQL queries: a COUNT(*) for the total, a GROUP BY sport for per-sport counts, and ORDER BY id DESC LIMIT 20 for the recent-interactions list. All results are returned as a plain PHP array.
  5. api.php JSON-encodes the array and sends it back. stats.html updates the four stat boxes and the recent interactions table directly in the DOM, no page reload required.

▶ Layer responsibilities

LayerFileResponsibility
Router api.php Reads HTTP method + action query param, routes to the correct controller method, sets response headers, JSON-encodes the return value. No business logic.
Controller application/controller/controller.php Input validation via strict allowlists (prevents SQL injection at the application level on top of prepared statements). Calls the Model and returns plain arrays, it never touches HTTP or SQL directly.
Model application/model/model.php All PDO/SQLite access lives here. Uses PDO::ATTR_ERRMODE_EXCEPTION for error propagation and bindParam prepared statements for every write. No business logic, pure data access.
View pages/stats.html Fetches JSON from the API via AJAX, updates the DOM. Polling is paused via the Page Visibility API when the tab is hidden and resumed on focus, avoids unnecessary requests in the background.
