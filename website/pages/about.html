About

I'm a sports fan, so when I had to pick a subject for my Web 3D Application I went with the three sports I actually enjoy watching and playing: tennis, basketball and football. The result is Sports 3D: a site that brings each of those sports into the browser as an interactive 3D scene you can rotate, zoom around and trigger animations on. The fun of it for me was trying to capture a bit of what makes each sport feel distinctive (the openness of a tennis court, the high-key arena lighting of a basketball game, the outdoor daylight of a football pitch) rather than building three identical scenes with different markings.

The 3D Models

I built all three scenes in Blender 5.1, modelling each one to follow the regulation dimensions of the real sport.

  • Geometry: Blender primitives only; playing surfaces, markings, nets, posts and stands kept as separate meshes so each component can have its own material and be toggled independently.
  • Lighting: added in the Three.js scene per page rather than baked into the GLB, so each sport has a lighting style that matches its mood (see the table above for the per-sport breakdown).
  • Cameras: each sport page has a small set of preset cameras (e.g. broadcast, low-angle, top-down) that the user can swap between. The football page also reads the camera transform straight from the Blender camera node in the GLB to keep framing consistent with the source file.
  • Materials: solid-colour PBR materials per component. Procedural shaders are avoided because glTF only exports image textures + base PBR, so anything I built with Voronoi/Noise gets simplified at export. Only image-textured or flat-PBR materials survive the round-trip.
  • Animations: keyframed in Blender's timeline at 30 fps with Bezier auto-clamped interpolation for smooth parabolic motion. The mixer in each page plays the GLB clip directly, no JS keyframes.
  • Render: ACES filmic tone mapping applied in the Three.js renderer for a more cinematic look. Each GLB is self-contained.

Design & Responsiveness

Usability & look and feel

  • Bootstrap 5.3 fluid grid so columns reflow predictably across screen sizes.
  • Consistent navbar on every page: user is always one click from anywhere on the site.
  • Anton display font (Google Fonts, SIL OFL) loaded as sports.ttf for headings gives the four pages a single visual identity.
  • Per-sport bloom intensity tuning so users can tell at a glance which scene they're in.

Responsiveness

Two iterations were needed to get this right on the sport pages.

  • Problem 1: The right-hand controls column on tennis/basketball/football didn't appear at all on mobile; the side panel was simply hidden below 700 px, leaving users with the 3D court but no way to trigger animations or switch cameras.
  • Fix 1: Rebuilt the side panel as a slide-in drawer triggered by a floating ☰ Controls button (see js/drawer.js); the 3D scene keeps the full viewport and controls are summoned on demand.
  • Problem 2: Tapping an animation opened the drawer but then the panel covered the scene, so users had to manually close the drawer before they could see the animation play.
  • Fix 2: Added a click listener that auto-closes the drawer when an animation or camera button is tapped, but deliberately leaves it open for scene-control toggles (wireframe, bloom, audio, lighting) so users can flip several in a row. Gated to <700 px so desktop is unaffected.

I tested all of this at 430 px (iPhone 14 Pro Max), 820 px (iPad Air) and 1920 px (desktop) using Chrome DevTools' Device Toolbar, plus on macOS Chrome and Safari at desktop width.

Media Integration

All six media types from the checklist are integrated:

  • 3D models: one .glb per sport, exported from Blender with cameras and animations enabled.
  • Video: functional_demo.mp4 screen-recording demonstrating each sport's interactions.
  • Audio: sport-specific sound effects fired at exact animation frames (kick at football frame 22, swish at basketball frame 46, racket-hit at tennis frame 11, etc.), gated behind an explicit toggle so the page never autoplays sound.
  • Images: the hero photo across the landing and About pages, optimised from 2.8 MB down to 433 KB after Lighthouse flagged it.
  • Text: descriptive content in the side panels of each sport page.
  • Animations: Blender-authored (tennis racket swing + ball bounce, basketball shot/dribble, football scoring run), triggered by button presses with frame-synced audio cues.

Interaction

Users interact with the site at three levels:

  • Manipulate: OrbitControls allow rotate / zoom / pan via mouse, trackpad or touch on every 3D scene.
  • View: four named camera presets per sport, plus a wireframe toggle revealing the underlying geometry.
  • Other pages: the landing page has clickable sport cards routing to each scene; the dedicated Stats page shows live AJAX-polled interaction counts from the PHP/SQLite backend.

MVC Architecture

Data flow

When a user presses a button on a sport page, the page calls SportsAPI.recordInteraction(sport, type) from js/dataLoader.js. That function POSTs a JSON body to api.php. The Controller receives the request, validates the sport value against the allowlist ['tennis','basketball','football'] and the type value against ['hit','juggle','shoot','dribble','swing'], then delegates to the Model. The Model runs a PDO prepared statement and writes a row to the SQLite database. On the read side, the Stats page calls SportsAPI.loadStats() every 5 seconds via a setInterval; the Controller calls the Model's aggregate query methods, serialises the result as JSON, and the Stats page updates the DOM stat boxes and table without a full page reload.

The three layers

Model (application/model/model.php): handles all PDO/SQLite access; opens the database connection in the constructor using the path __DIR__ . '/../../db/sports.db'; provides four public methods: dbCreateTable(), dbInsertData(), dbGetData(), and dbGetStats(). It contains no business logic, no HTTP-layer code, and no input validation. It is a pure data-access layer.

Controller (application/controller/controller.php): validates the incoming sport field against ['tennis','basketball','football'] and the type field against ['hit','juggle','shoot','dribble','swing'] before delegating any operation to the Model. It returns plain PHP arrays to the API router; it never touches HTTP headers or writes SQL directly.

View (stats.html + application/view/viewStats.php): reads JSON from the API endpoint; updates the DOM stat boxes and interaction table on each poll cycle. When the tab is hidden the Page Visibility API (document.addEventListener('visibilitychange'...)) pauses polling to avoid unnecessary server requests, resuming when the tab becomes visible again.

Key code snippet: Model constructor and dbInsertData

PHP application/model/model.php 
class Model {
    private PDO $db;

    public function __construct() {
        $dbPath = __DIR__ . '/../../db/sports.db';
        $this->db = new PDO('sqlite:' . $dbPath);
        $this->db->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);
        $this->db->setAttribute(PDO::ATTR_DEFAULT_FETCH_MODE, PDO::FETCH_ASSOC);
    }

    public function dbInsertData(string $sport, string $type): int {
        $stmt = $this->db->prepare(
            "INSERT INTO interactions (sport, type) VALUES (:sport, :type)"
        );
        $stmt->bindParam(':sport', $sport);
        $stmt->bindParam(':type',  $type);
        $stmt->execute();
        return (int) $this->db->lastInsertId();
    }
}

Dual-backend pattern

The front-end supports two backend patterns in the same codebase. js/dataLoader.js tries the PHP/SQLite endpoint first (Lab 7/8 pattern) and automatically falls back to a static data.json file (Lab 6 pattern) if the server is unavailable. A console log on every page load reports which path is active ([Data Loader] Loaded from PHP/SQLite backend or [Data Loader] PHP unavailable, using JSON fallback), verifiable in DevTools during marking without changing any code.

Why I Chose These Technologies

  • Three.js: the WebGL library taught and demonstrated throughout the labs. It handles GLB loading, AnimationMixer, EffectComposer, and OrbitControls out of the box, avoiding the need to write raw WebGL shader code.
  • Bootstrap 5.3: the responsive grid framework used in the module labs. Its fluid column system and utility classes handle breakpoint reflow predictably, and its accessibility primitives (landmark roles, contrast defaults) reduce manual WCAG work.
  • Blender 5.1: the 3D authoring tool taught in the module. Its Python bpy API lets you drive and automate the scene; it exports cleanly to GLB with cameras, animations, and PBR materials intact.
  • PHP + SQLite: the server-side stack covered in Labs 7–8. SQLite requires no separate database server and runs natively on the university's hosting environment; PDO with prepared statements gives parameterised queries without adding a heavy ORM dependency.
  • Native fetch() instead of jQuery: the project avoids jQuery because the modern Fetch API covers all AJAX requirements (posting JSON, error handling, polling) without an extra 87 KB dependency. The module introduced jQuery as an option but fetch() is now the browser standard.

Going Beyond the Labs

I went beyond the lab content in six ways:

1. Frame-triggered audio. Each sport page fires sound effects at exact animation frames using a triggers array, for example the basketball swish plays when the shot animation reaches frame 46, the football kick fires at frame 22 and the crowd cheer follows at frame 60. A played Set prevents re-firing within a playthrough, and the system integrates with the AudioContext gesture requirement via an explicit toggle so it satisfies browser autoplay policies.

2. Post-processing pipeline. All scenes use Three.js EffectComposer chaining a RenderPass → UnrealBloomPass → OutputPass. Bloom strength is tuned per sport (0.35 for football to preserve outdoor daylight; 0.5 for basketball for arena high-key lighting), with a runtime toggle to disable it.

3. Custom REST API with MVC architecture. The backend follows the Lab 7 MVC pattern, model.php handles PDO/SQLite with prepared statements, controller.php validates input against allowlists, viewStats.php renders a server-side Bootstrap table, and api.php is a thin (<90 line) router. A live demo is available on the Stats page, every interaction on the sport pages POSTs to the API, and the Stats page polls it back over AJAX every five seconds.

4. Basketball surface selector with localStorage persistence. The basketball court GLB contains two floor meshes (Floor_Wood and Floor_Concrete) occupying the same position in the scene. A surface-selector panel in the side panel lets users toggle between them at runtime by flipping each mesh's .visible flag, so only one is ever rendered and there is no z-fighting. The choice is persisted to localStorage under the key bball.surface and restored on next page load. The meshes are located by name after the GLTFLoader callback resolves, with a graceful console.warn if either is missing.

5. Mobile drawer pattern. js/drawer.js is a self-contained IIFE that injects the hamburger toggle and slide-in panel at runtime, with ESC dismissal, auto-close on selection and aria-expanded state for screen readers. Activates only below 700 px so desktop layout is unchanged.

6. Dual-backend pattern (Lab 6 + Lab 7/8 in one codebase). js/dataLoader.js is a shared IIFE that wraps every API call site. On read it tries the PHP/SQLite endpoint first; if the server returns a non-2xx status, network error, or malformed JSON it falls back to data.json automatically. On write it no-ops silently in fallback mode rather than throwing. A console log on every page load ([Data Loader] Loaded from PHP/SQLite backend or [Data Loader] PHP unavailable, using JSON fallback) lets the marker verify which path is active in DevTools without changing any code. This demonstrates both the Lab 6 JSON pattern and the Lab 7/8 PHP+SQLite pattern in the same front-end.

Accessibility

Accessibility was considered at three levels: page structure, interactive controls, and visual contrast. The site was built mobile-first and validated across the tools listed below.

Page structure & semantics

  • Every page wraps its main content in a <main> landmark and uses semantic HTML5 sectioning (<nav> for the top navigation, <header> for page intros, <section> for grouped content, and <footer> for credits) so screen readers can navigate the document outline without relying on visual cues.
  • Every <img> tag carries descriptive alt text, including the wireframe screenshots, Lighthouse audit captures, and the per-sport scene photographs.
  • The hamburger drawer button on mobile (js/drawer.js) sets aria-expanded="true|false" as it opens and closes, so assistive technology announces the state change.

Keyboard & interactive controls

  • All camera, animation, lighting, bloom, audio, and wireframe controls are real <button> elements rather than styled <div>s, so they are reachable via Tab order and activatable with Enter or Space without any custom keybinding code.
  • Visible focus rings are preserved (no outline: none reset), so keyboard users always know which control has focus.
  • The mobile drawer can be dismissed with the Escape key as well as the close button, matching standard modal dismissal patterns.

Visual contrast & alternative views

  • Body text and form fields hit WCAG 2.1 AA contrast ratios: verified against the Chrome Lighthouse Accessibility audit (final mobile score 91).
  • The wireframe toggle on every sport page gives a high-contrast line-art view of the geometry as an alternative to the shaded render, useful both as a visual style toggle and as a fallback view if the lit scene is hard to parse.
  • Audio is opt-in only via an explicit toggle, so users with hearing impairments, in quiet environments, or using screen readers are not interrupted by autoplaying sound effects.

Tested viewport range

  • Validated at 430 px (iPhone 14 Pro Max), 820 px (iPad Air) and 1920 px (desktop) using Chrome DevTools' Device Toolbar, plus on macOS Chrome and macOS Safari at desktop width. No horizontal scrolling, no clipped content, and no inaccessible controls were observed at any breakpoint.

User Feedback

Three external testers stepped through the site unguided as part of a cognitive walkthrough. Each was given the single instruction 'look around this site and try one of the sports' and observed without prompting. Their feedback drove three concrete changes to the build.

TesterObservationAction taken
Tester A Said the navbar "Back" button was confusing, wasn't clear whether it meant browser-back or return to the home page Renamed it to Home on every page so the destination is unambiguous
Tester B Reported that the basketball scene was hard to make out, the page background was solid black so dark parts of the model blended into it Changed the basketball scene background to a lighter sky-blue tone so the court, hoop and stanchion are clearly visible against it
Tester C Suggested adding a goalkeeper to the football pitch and either a ball-return / opposing player on the tennis court, plus some form of scoring target on both Acknowledged as a valid enhancement but out of scope for this submission. Noted as a future iteration. Both scenes are already structured (separate meshes per component, AnimationMixer wired up) so a keeper or target model could be dropped in without re-architecting anything
